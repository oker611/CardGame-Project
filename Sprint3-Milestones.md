# 第三周开发迭代对齐计划
## 第一次对齐：周四晚
**目标**：各模块接上主干，跑通“最小闭环”

### 必须交付
1. **代码结构到位**
至少完成并提交以下核心类/接口骨架：
- Card
- Player
- GameState
- GameEngine
- RuleEngine
- GameController
- GameActionHandler

2. **命名与接口冻结**
仓库中必须已经存在并统一使用：
- docs/Naming-Convention.md
- docs/Interface-Contract.md

要求：
- 之后不再随意改核心类名、方法名、DTO字段名
- UI、逻辑、规则三方按同一套接口开发

3. **最小可运行原型**
必须至少实现下面这个最小流程中的大部分：
初始化游戏 -> 发牌 -> 确定首出玩家 -> 展示当前玩家和手牌 -> 可以选择牌 -> 点击出牌按钮有响应

“有响应”最低标准：
- 按钮能触发逻辑调用
- 能打印日志或弹出提示
- 不要求完整规则已全部实现

4. **阶段证明材料**
群里或仓库必须能看到：
- 运行截图 / 录屏
- 当前代码目录结构
- 已实现模块说明

**P.S.**
- UI必须通过 GameActionHandler 调用逻辑，不允许直接修改 GameState

### 周四验收标准
满足以下 4 条算通过：
- 能启动项目
- 能进入游戏主页面或测试入口
- 发牌逻辑基本可跑
- 出牌按钮已经和逻辑层接通，不是空壳

---

## 第二次对齐：周六晚
**目标**：跑通“一局简化游戏原型”

### 必须交付
1. **核心规则初稿可用**
至少支持：
- 单张
- 对子
- 首轮方块3先出
- 压过上家
- Pass

要求：
- validatePlay() 可实际工作
- 对合法/非法出牌有明确反馈

2. **主流程可跑通**
必须能跑通这一整条链路：
开始游戏 -> 发牌 -> 当前玩家选牌 -> 出牌/过牌 -> 系统校验 -> 更新桌面状态 -> 切换下一玩家 -> 重复若干轮 -> 有人出完后结束

3. **UI 原型升级到“可操作”**
UI 至少包含：
- 当前玩家显示
- 手牌区
- 出牌按钮
- Pass 按钮
- 上一手牌/提示信息区

要求：
- 交互不一定美观，但必须能操作
- 不接受只画界面不接逻辑

4. **日志与调试信息**
必须有清晰输出，至少包括：
- 发牌结果
- 当前轮到谁
- 玩家出牌结果
- 是否通过校验
- 是否Pass
- 谁获胜

5. **阶段 Bug 清单**
需要整理一个简短问题清单，例如：
- 目前规则只支持到什么程度
- UI 还缺什么
- 哪些地方是已知 bug

### 周六验收标准
满足以下 5 条算通过：
- 可以实际操作一局简化游戏
- 出牌和 Pass 都能工作
- 游戏状态会更新
- 有结束条件
- 有可展示的运行结果

---

## 第三次对齐：下周二（第四次迭代开始点）
**目标**：完成 Sprint 3 收口，给 Sprint 4 提供稳定基线

本次重点：判断哪些内容稳定可继承到第4周，哪些需要重构/补做

### 必须交付
1. **稳定版本基线**
仓库中必须有一个可回退、可演示的稳定版本，要求：
- main 分支或指定分支可运行
- 能复现周六的简化游戏原型
- 不依赖某个人本地未提交代码

2. **Sprint 3 最终交付清单**
需要明确写出：
- 已完成模块
- 未完成模块
- 半完成模块
- 当前规则范围
- 当前 UI 范围

建议形成文档：
- docs/Sprint3-Review.md

3. **模块完成状态表**
至少对以下模块逐项说明状态：
- 游戏逻辑
- 规则系统
- UI
- 控制层接口
- AI 占位
- 联机接口占位
- 文档与命名规范

状态统一为：
- 已完成
- 可运行但待优化
- 未完成

4. **Sprint 4 输入材料**
必须准备好下一迭代继续开发所需内容：
- 当前代码架构
- 当前接口文档
- 当前 bug 清单
- 下一周优先级列表

5. **风险收敛结论**
需明确说明：
- 现在系统最稳的部分是什么
- 现在最危险的部分是什么
- 第四周先补哪里最值

### 下周二验收标准
满足以下 5 条算通过：
- 有稳定可运行版本
- 有 Sprint 3 总结文档
- 有明确 bug/缺口列表
- 有 Sprint 4 优先级
- 团队所有人知道自己接下来做什么
